# [FIRE_KIDS] 問い合わせ管理パイプライン v2 導入手順書

作成日: 2026-05-20
対象: v1 既稼働の Apps Script プロジェクトに `Code.v2.gs` を追加して P1〜P5 を有効化する手順
所要時間: 約30分（Webhook作成・ラベル整理を含む）

---

## 0. 前提

- v1（`Code.gs`）が既に動作している
- スクリプトプロパティに `SHEET_ID`／`SLACK_WEBHOOK_URL` が設定済み
- Gmail ラベル `問い合わせ-未処理` / `問い合わせ-処理済` が作成済み

---

## 1. 広告実績チャネル用 Webhook の準備（任意）

広告実績チームのSlackチャネルへ朝会サマリを同報する場合のみ実施。

1. Slack ワークスペース → アプリ管理 → **Incoming Webhook** を追加
2. 投稿先チャンネルを広告実績チームのチャネル（例: `#ad-performance`）に指定
3. 発行された Webhook URL を控える（`https://hooks.slack.com/services/...` 形式）

> 同報しない場合はこの手順をスキップしてOK。後から追加可能。

---

## 2. ファイル配置

Apps Script プロジェクトに以下のファイルを追加（コピペ）:

| ファイル名 | 配置先 | 役割 |
|---|---|---|
| `Code.v2.gs` | Apps Script エディタ | v2 本体（v1と共存） |

`Code.gs`（v1）は **削除しない**。両方残した状態で v2 が v1 を引き継ぐ。

---

## 3. スクリプトプロパティ追加

Apps Script エディタ → 歯車 → **スクリプトプロパティ** で以下を追加。

| キー | 値 | 必須 |
|---|---|---|
| `SHEET_ID` | （v1から継承） | ✓ |
| `SLACK_WEBHOOK_URL` | （v1から継承。メインチャネル） | ✓ |
| `SLACK_WEBHOOK_URL_ADPERF` | 手順1で発行した広告実績用 Webhook URL | 任意 |
| `ADMIN_EMAIL` | エラー通知先メール | 任意（**P1.5ヒートビート用に推奨**） |
| `DRIVE_ATTACHMENT_FOLDER_ID` | 添付保存先フォルダID | 任意（未設定なら自動作成） |
| `ORG_DOMAIN` | 社内メールドメイン（例: `firekids.jp`）| **P1.5: Gmail返信自動検知用**。未設定なら検知無効 |
| `BODY_VIEWERS` | `inquiry_bodies` を閲覧・編集できる追加メール（カンマ区切り）| **P1.6-4: 本文保護用**。所有者以外に開示する人のみ列挙 |
| `NEXT_ID` | （GASが自動管理） | 触らない。setupV2 が初期化 |

---

## 4. setupV2() を実行

1. Apps Script エディタで `Code.v2.gs` を開く
2. 関数選択ドロップダウンから `setupV2` を選択 → ▶ 実行
3. 初回は権限承認ダイアログが出るので、**全て許可**
4. 完了すると `setupV2 完了` のログが出る

### この実行で起きること
- `inquiries` シートに **6列を末尾追加**（カテゴリ／優先度／SLA期限／初回応答日時／添付／リマインド履歴）
- `_config` シートにカテゴリ辞書を投入（不足ヘッダのみ補完）
- `_history` / `inquiry_bodies` / `_ingest_rules` / **`_templates`** シートを新規作成
- 添付保存先 Drive フォルダ `firekids_inquiry_attachments` を自動作成
- v1 のトリガー（`fetchAndSync`／`onSheetEdit`）を**削除**し、v2 のトリガー8本を登録:
  - `fetchAndSyncV2` — 5分毎（取り込み専念）
  - `onSheetEditV2` — シート編集時
  - `remindOverdue` — 1時間毎
  - `postDailySummary` — 毎日09:00（平日のみ実送信）
  - `staleCheck` — 毎日12:00（長期放置検知）
  - `heartbeat` — 毎日08時台（パイプライン停止検知）
  - `detectFirstRepliesJob` — 30分毎（P1.6-2: Gmail返信自動検知）
  - `rotateHistoryJob` — 毎週日曜03時台（P1.6-3: `_history` ローテーション）
- `_history_archive` シートを新規作成（隠し）
- `NEXT_ID` スクリプトプロパティを初期化（P1.6-1: ID採番カウンタ）
- `inquiry_bodies` シートを保護（P1.6-4: 編集権限を所有者＋`BODY_VIEWERS` 限定）

---

## 5. 取り込み対象アドレスの設定（**最重要**）

v2 のメイン取り込み方式は **シートで管理するホワイトリスト**。Gmailフィルタを触らずに、シート上で「この From/To のメールを取り込む」を宣言する。

### 5.1 `_ingest_rules` シートを開く
`setupV2()` 実行後、スプレッドシートに `_ingest_rules` シートが自動作成されている（サンプル3行は **無効状態**で配置済み）。

### 5.2 取り込みルールを追加・有効化
列の意味:

| 列 | 意味 | 例 |
|---|---|---|
| From条件 | 送信元アドレス（部分一致、複数は `OR` 区切り）| `noreply@form.firekids.jp` ／ `taro@example.com OR jiro@example.com` |
| To条件 | 受信先アドレス | `info@firekids.jp` |
| 件名条件(正規表現) | 件名フィルタ。空欄なら全件OK | `査定\|買取\|見積` |
| 既定カテゴリ | このルール一致時に自動付与するカテゴリ | `査定・買取` |
| 既定優先度 | 同じく優先度 | `高` |
| 有効 | チェックボックス | TRUE で取り込み開始 |

**設定の典型例**:

| From条件 | To条件 | 件名条件 | 既定カテゴリ | 既定優先度 | 有効 |
|---|---|---|---|---|---|
| `noreply@form.firekids.jp` |  |  | フォーム | 中 | TRUE |
|  | `info@firekids.jp` |  |  |  | TRUE |
|  | `support@firekids.jp` | `査定\|買取\|見積` | 査定・買取 | 高 | TRUE |
| `noreply@yahoo.co.jp` |  |  | 一般問い合わせ | 低 | TRUE |

### 5.3 動作の仕組み
- `fetchAndSyncV2` 実行時、各有効ルールを Gmail検索クエリ `from:(...) to:(...) newer_than:30d -label:問い合わせ-処理済` に変換
- ヒットしたメールをシートへ追記し、`問い合わせ-処理済` ラベルを付与（重複防止）
- **メアド追加だけで取り込み開始**。Gmailフィルタの作成は不要

### 5.4 ラベル方式（後方互換・任意）

旧運用や手動振り分けを併用したい場合のみ。子ラベルがあれば**カテゴリ列に自動反映**される:
```
問い合わせ-未処理            ← 親
問い合わせ-未処理/フォーム   ← 子（カテゴリ列に「フォーム」が入る）
問い合わせ-未処理/メール直   ← 子
```

> ホワイトリスト方式とラベル方式は **併用可**。Gmail Message ID で重複取り込みは抑止される。

## 6. 動作確認

### 6.1 ヘルスチェック
シートを開いて `問い合わせパイプライン v2` メニュー → `ヘルスチェック (v2)` を実行。
全項目に ✓ が付くことを確認。

### 6.2 Slack 疎通テスト
同メニュー → `Slack通知テスト (v2)` を実行。メインチャネルにテストメッセージが届く。

### 6.3 デイリーサマリ手動送信
同メニュー → `デイリーサマリを今すぐ送信` を実行（平日のみ実送信）。
内容を確認して、フォーマットがチームに合うか確認。

### 6.4 メール取得テスト（ホワイトリスト方式）
1. `_ingest_rules` シートの「有効」をひとつでも TRUE にする（例: From=自分のメアド）
2. 自分宛にテストメールを送る（条件にマッチする From/To で）
3. メニュー → `今すぐメール取得 (v2)` を実行
4. `inquiries` シートに新規行が追加され、Slack通知が届くことを確認
5. Gmail 側でメールに `問い合わせ-処理済` ラベルが付くことを確認

### 6.5 メール取得テスト（ラベル方式・任意）
ラベル方式を併用する場合のみ。Gmail で `問い合わせ-未処理` ラベルを付与 → メニューから取り込み実行。

## 6.6 P1.5（ミス防止＆工数削減）の動作確認

### 6.6.1 完了時メモ必須化（A1）
1. テスト行のステータスを「完了」に変更（メモは空のまま）
2. → ステータスが自動で「対応中」に戻り、Slack に警告が届けばOK
3. メモを書いてから再度「完了」に変更すれば、今度は通過する

### 6.6.2 長期放置検知（A2）
- 手動で動作確認したい場合は、メニュー → `長期放置チェックを今すぐ実行`
- 本番は毎日12:00 に自動で動き、3日以上動いていない「対応中」案件があれば Slack通知

### 6.6.3 ヒートビート（A4）
- `ADMIN_EMAIL` を設定しておくこと
- メニュー → `ヒートビートを今すぐ確認`（直近12hで取得が走っていなければ管理者宛メール＋Slack警告）

### 6.6.4 Gmail返信自動検知（A5）
1. `ORG_DOMAIN` をスクリプトプロパティに設定（例: `firekids.jp`）
2. テスト案件の Gmail スレッドから、社内ドメイン宛から返信を送る
3. 次回 `fetchAndSyncV2` 実行時に、対応中の `初回応答日時` が自動セット／ステータスも未対応→対応中へ昇格

### 6.6.5 対応期限自動セット（B2）
- 新規受信時に「対応期限」が「SLA期限」と同じ値で自動入力される
- 例外的に期限を延ばしたい場合は、運用者が「対応期限」だけ手動で上書き

### 6.6.6 返信下書き自動生成（B3）
1. `_templates` シートに5パターン（査定・修理・在庫・クレーム・共通）が投入済み。必要に応じて文体を編集
2. `inquiries` シートで対象行を選択
3. メニュー → `選択行から返信下書きを作成`
4. Gmail に返信ドラフトが自動作成される（プレースホルダ `{name}` `{id}` `{subject}` は自動展開）

### プレースホルダ仕様（`_templates`）
| 変数 | 展開内容 |
|---|---|
| `{name}` | From ヘッダから抽出した名前（無ければローカルパート） |
| `{id}` | 管理ID（inquiries シートの ID 列） |
| `{subject}` | 元メールの件名 |

## 6.7 P1.6（堅牢化）の動作確認

### 6.7.1 ID採番（6-1）
- ヘルスチェックで `NEXT_ID` が数値で表示されること
- 行を削除しても次の取り込みで重複IDが出ないこと

### 6.7.2 タイムアウト分離（6-2）
- Apps Script → トリガー一覧で `detectFirstRepliesJob`（30分毎）が登録済みであること
- `fetchAndSyncV2` の実行時間が安定して 1〜2分以内に収まること（実行履歴で確認）

### 6.7.3 履歴アーカイブ（6-3）
- 初期は `_history_archive` は空（90日経つまで何も入らない）
- 手動で動作確認する場合は Apps Script エディタから `rotateHistoryJob` を直接実行

### 6.7.4 本文保護＋PIIマスク（6-4）
1. `BODY_VIEWERS` プロパティに、本文を閲覧する必要があるメンバーのメアドをカンマ区切りで設定
2. `setupV2()` を再実行 → `inquiry_bodies` シートに「保護されたシート」のマーク（鍵アイコン）が付く
3. 設定外のユーザーがシートを開いても本文行を編集できないこと
4. 本文中の電話番号・カード番号風文字列が `[PHONE-REDACTED]` / `[CARD-REDACTED]` に置換されていること

## 7. ロールバック手順（不具合発生時）

1. Apps Script → トリガー一覧 → v2 のトリガー4本を全削除
2. v1 のトリガーを手動で復元（`fetchAndSync` 5分毎／`onSheetEdit` onEdit）
3. シート末尾に追加された6列は **削除しなくてもv1動作に支障なし**（v1は無視するため）

v2 を完全に消したい場合は `Code.v2.gs` ファイルを削除すればOK。

---

## 8. 運用開始後の見直しポイント（2週間後）

- 自動分類の誤判定が多いカテゴリ → `_config` シートの **キーワード列**を編集（カンマ区切り）
- リマインド通知が多すぎる／少なすぎる → `Code.v2.gs` の `V2_REMIND_UNHANDLED_HOURS` / `V2_REMIND_URGENT_HOURS` を調整
- デイリーサマリのフォーマット改善 → `buildDailySummaryText_` を編集
- 期待効果が確認できたら P6（Slack ID マッピング）／P7（Bot Token化）／P8（KPIダッシュボード）の追加実装を検討

## 9. トラブルシューティング

| 症状 | 対処 |
|---|---|
| `SHEET_ID または SLACK_WEBHOOK_URL が未設定` | スクリプトプロパティを再確認 |
| 添付ファイルがDriveに保存されない | `DRIVE_ATTACHMENT_FOLDER_ID` をクリアして `setupV2` 再実行（自動作成に戻る） |
| デイリーサマリが届かない | 平日09:00以降に確認／メニューから手動実行で疎通確認 |
| リマインドが同じ件で連発する | `inquiries` シートの「リマインド履歴」列（最右）の JSON を確認。`hasRecentReminder_` の窓は24h |
| カテゴリが「一般問い合わせ」ばかり | `_config` シートのキーワード辞書を見直し（運用2週間で精度向上を狙う） |
| ホワイトリストに登録したのに取り込まれない | (a) `有効` が TRUE か (b) `From条件` の綴り (c) Gmailで該当メールが `問い合わせ-処理済` ラベル付きになっていないか確認 |
| 過去30日より前のメールを取り込みたい | `Code.v2.gs` の `V2_INGEST_LOOKBACK` を `'90d'` 等に変更して `setupV2` 再実行 |
| 完了に変えても自動で対応中に戻ってしまう | 対応メモが空欄。メモを書いてから完了にする（A1の意図された挙動） |
| 返信下書き作成でエラー | (a) 対象行のカテゴリと一致するテンプレが `_templates` にあるか (b) Gmail Message ID 列が空でないか |
| Gmail返信自動検知が動かない | `ORG_DOMAIN` が設定されているか確認 |
| ヒートビート警告が誤発火 | 5分毎の `fetchAndSyncV2` トリガーが生きているか確認。Apps Script の実行履歴も確認 |
| ID列が空 or 既存より小さい数値 | スクリプトプロパティ `NEXT_ID` を確認。0や空なら `setupV2()` 再実行で既存最大ID+1から再開 |
| `_history_archive` シートが見つからない | 隠しシート。スプレッドシート → メニューバー → 全シート一覧 → 表示 |
| `inquiry_bodies` を編集できない | 意図された挙動（P1.6-4）。閲覧編集を許可するなら `BODY_VIEWERS` にメアドを追加して `setupV2()` 再実行 |
| 本文がマスクされすぎる／されない | `Code.v2.gs` の `V2_PII_PATTERNS` の正規表現を調整 |
