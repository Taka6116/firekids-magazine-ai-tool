## 1. ① メール転送→GAS→Sheets の強化

### 1.1 取り込みの2方式（並行運用）

v2 では **「シートで管理するホワイトリスト方式」** をメイン取り込み経路に格上げし、Gmailフィルタへの依存を最小化する。従来のラベル方式は後方互換として維持。

#### 方式A: ホワイトリスト方式（推奨・メイン）
- `_ingest_rules` シート に「取り込み対象」のルールを行として並べる:

| From条件 | To条件 | 件名条件(正規表現) | 既定カテゴリ | 既定優先度 | 有効 |
|---|---|---|---|---|---|
| `noreply@form.firekids.jp` | | | フォーム | 中 | TRUE |
| `*@gmail.com` | `info@firekids.jp` | `査定\|買取` | 査定・買取 | 高 | TRUE |
| | `support@firekids.jp` | | | | TRUE |

- GAS は各行を Gmail 検索クエリ `from:(...) to:(...) newer_than:30d -label:問い合わせ-処理済` に変換して取得
- ヒットしたメールをシートに追記し、`問い合わせ-処理済` ラベルを付与（重複防止）
- **メールアドレスを追加するだけで取り込み開始できる** → 運用担当者がGmailフィルタを触る必要がない

#### 方式B: Gmailラベル方式（v1からの後方互換）
- 親ラベル `問い合わせ-未処理` 配下の子ラベル（`/フォーム`／`/メール直` 等）を一括スキャン
- Gmailフィルタで人手で振り分けたい場合に使用
- 方式Aと**重複しても二重取り込みは発生しない**（Gmail Message ID で抑止）

#### 既定カテゴリの優先順位
取り込み時のカテゴリ決定は以下の優先度:
1. `_ingest_rules` の「既定カテゴリ」列に値があればそれを採用
2. Gmail子ラベル名（方式Bのとき）
3. キーワード自動分類（§1.2）の結果

### 1.2 自動分類（カテゴリ・優先度）
受信時に件名・本文から下記を判定し、シートに自動入力する。

| カテゴリ | 判定キーワード（件名 or 本文） | 優先度初期値 | SLA(目安) |
|---|---|---|---|
| 査定・買取 | 査定／買取／いくら／見積もり | **高** | 24h |
| 修理・OH | 修理／オーバーホール／止まる／動かない | **中** | 48h |
| 在庫問い合わせ | 在庫／取り置き／入荷 | **中** | 24h |
| クレーム | クレーム／不具合／返品／返金／弁護士 | **緊急** | 4h |
| 一般問い合わせ | 上記以外 | 低 | 72h |

判定ロジックは `classifyInquiry_(msg)` 関数として実装。優先度は手動上書き可。

### 1.3 本文全文保存
**現状の問題**: 本文を500字プレビューだけにしているため、長文問い合わせの全文確認はGmailを開く必要がある。

**強化案**:
- 専用シート `inquiry_bodies` を追加（列: `ID / Gmail Message ID / 本文全文 / HTML本文 / 受信日時`）
- メインの `inquiries` シートには引き続きプレビュー500字を表示
- 全文閲覧は **メインシートの「本文プレビュー」セルの注釈** or `inquiry_bodies` シートへのリンクで参照

### 1.4 添付ファイル保存
**強化案**:
- 添付があれば Google Drive のフォルダ `MAGAZINE/inquiry_attachments/{YYYY-MM}/` に保存
- ファイル名: `{受信日時}_{送信元ローカルパート}_{元ファイル名}`
- メインシートに新列 `添付` を追加し、Driveフォルダリンクを記録（複数ファイルはコンマ区切りリンク）

## 2. ② Sheets→Slack Webhook の強化

### 2.1 しきい値リマインド
時限トリガー `remindOverdue` を **毎時0分** で実行。

| 条件 | 通知文言 | 通知先 |
|---|---|---|
| `ステータス=未対応` かつ 受信から24h経過 | `:warning: 未対応24h経過` | 担当者（未アサインなら全員メンション） |
| `ステータス=未対応／対応中` かつ `対応期限<今日` | `:rotating_light: 期限超過` | 担当者＋管理者 |
| `優先度=緊急` かつ `ステータス=未対応` | 4h経過で即通知 | `@channel` |

**重複防止**: シートに `リマインド送信履歴` 列（タイムスタンプ配列をJSON保存）を持たせ、同条件で24h以内は再送しない。

### 2.2 担当者→Slack IDマッピング
- `_config` シートに `担当者名 / Slack Member ID / メールアドレス` の3列マッピングを保持
- Slack通知の本文で `<@U01ABCDE>` 形式メンションに変換
- 未登録の担当者はそのまま名前表示＋警告ログ

### 2.3 ステータス変更ログ（Webhook前提）
- Webhookでは `ts` が取得できないためスレッド返信は不可
- 代替として **`_history` シートに変更ログを記録**（誰が・いつ・どのID／どの列を／何から何へ変更したか）
- Slack再投稿は行わず、デイリーサマリで前日のステータス変動として集計

### 2.4 デイリーサマリ
時限トリガー `postDailySummary`、平日09:00 JST 実行。

```
:sunny: [FIRE_KIDS] 朝会用 問い合わせサマリ (2026-05-19 月)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
■ 昨日の動き
  新着: 12件 / 完了: 8件 / 残:  4件
■ 現在のステータス
  未対応:  6件 (うち優先度高2件)
  対応中:  3件
  保留 :  1件
■ 期限アラート
  期限超過: 2件 → @sasaki, @tanaka
  本日期限: 4件
■ 担当別未完件数
  sasaki: 3 / tanaka: 4 / yamada: 2 / 未アサイン: 1
[シートを開く](https://docs.google.com/spreadsheets/d/.../edit)
```

## 4. ④ 広告実績チーム朝会オペレーション統合

### 4.1 朝会フロー（広告実績チームのデイリーオペに乗せる）

```
平日09:00 ─┬─ GAS postDailySummary 実行
          ├─ Slack #ad-performance に投稿
          ├─ Slack #firekids-inquiry にも投稿（本チャネル）
          └─ シート _dashboard を最新化
↓
09:05〜  朝会内で広告実績チームが下記を確認:
   1. 期限超過の件数（あればその場で担当割り当て or エスカレ）
   2. 当日期限の件数（担当者の稼働確認）
   3. 緊急優先度の有無
↓
09:30   担当者は自分の未対応分を着手 → ステータスを「対応中」へ
```

### 4.2 朝会チェックリスト（Webhookプレーンテキスト）
デイリーサマリ末尾に **プレーンテキストの確認項目** を埋め込む（Webhook制約のためチェックボックスUIは未使用）:

```
■ 朝会チェック
  ・期限超過2件、対応者をその場で確定したか
  ・緊急優先度の有無を確認したか
  ・未アサイン1件を引き取ったか
  ・前日完了8件のうち、再発対応が必要なものはないか
```

### 4.3 「漏れ防止」の二重化
- **広告実績チームは毎朝必ずこのサマリを見る** → 担当外でも未対応・期限超過に気づける
- 期限超過が **2営業日以上連続で残った場合**、管理者宛に追加メール通知（`escalateLongOverdue` 関数）
- 完了せず1週間以上経過したら自動で「保留」ステータスへ遷移＋管理者通知

## 4.5. Phase P1.5: ミス防止＆工数削減セット

P1〜P5 完了後に**ほぼ同等の重要度**で追加実装する。運用負荷をさらに下げ、漏れ・放置・属人化を機械的に潰す層。

### A. ミス徹底防止
| ID | 仕組み | 実装位置 |
|---|---|---|
| A1 | **完了時の対応メモ必須化** — `ステータス=完了` に変更しようとして `対応メモ` が空ならステータスを元に戻し、Slackで警告 | `onSheetEditV2` |
| A2 | **長期放置検知** — `対応中` のまま3日以上「最終更新（`_history` 最終行）」が動かない案件を毎日12:00 にSlackリスト化 | `staleCheck` 関数 / 時限トリガー |
| A4 | **ヒートビート監視** — `fetchAndSyncV2` 実行のたびに `LAST_FETCH_TS` を更新し、毎朝08:55 に12h以上動いていなければ管理者へエラーメール | `heartbeat` 関数 / 時限トリガー |
| A5 | **Gmail返信自動検知** — 取り込み時に各未完案件のスレッドを覗き、社内ドメイン送信が含まれていれば「初回応答日時」自動セット＋ステータス未対応なら「対応中」へ昇格 | `detectFirstReplies_` / `fetchAndSyncV2` から呼出 |

### B. 工数削減
| ID | 仕組み | 実装位置 |
|---|---|---|
| B2 | **対応期限の自動セット** — 受信時に SLA期限 と同じ値を `対応期限` 列へ自動転記。SLAから外れた期限を入れたい時だけ運用者が上書き | `appendInquiryRowV2_` |
| B3 | **返信テンプレ自動下書き** — `_templates` シート（カテゴリ／件名テンプレ／本文テンプレ）を新設。シート上で行選択 → メニューから1クリックで Gmail に下書きを自動生成 | `_templates` / `createReplyDraftForRow_` |

> 担当者の割当は人手で行う方針（自動アサインは採用しない）。

### 新規シート
- `_templates` — カラム: `カテゴリ / 件名テンプレ / 本文テンプレ`。`{name}` `{subject}` `{id}` などプレースホルダ対応

### 新規スクリプトプロパティ
- `LAST_FETCH_TS` — ヒートビート用（GAS が自動更新）
- `ORG_DOMAIN` — 社内ドメイン（A5判定で `from` がこのドメインなら自社送信扱い）。例: `firekids.jp`

### 新規トリガー
- `staleCheck` — 毎日12:00
- `heartbeat` — 毎日08:55

## 6. 確認事項・既知の制約

### 制約1: Webhookでは下記が不可（割り切り）
- ステータス変更時のスレッド返信
- `<@USER>` 形式の確実なメンション（IDが取得できないため）
- Slack `checkboxes` ブロック要素

→ いずれも v2 では実装しない（§2.3／§4.2 参照）。必要になった段階でP7のBot Token化で対応。

### 制約2: 広告実績Webhookのチャネル
- 広告実績チーム側のチャネルに **Incoming Webhook を追加** してもらい、その URL を スクリプトプロパティ `SLACK_WEBHOOK_URL_ADPERF` に登録する
- 未設定なら同報をスキップし、メインチャネルのみへ投稿（フォールバック）

### 制約3: 自動分類の精度
キーワードベースは取りこぼし・誤分類が一定発生する。運用2週間で誤判定をログ収集し、`_config` シートの **キーワード辞書** を更新する運用とする。AI分類（GPT/Claude API）はコストが読めないため初期は不採用。

### 制約4: 既存運用との互換性
v1で既に運用中の場合、列追加はシート末尾に追加するだけにし、v1 `COL` 定数の番号は維持して既存データを壊さない。`setupV2()` 関数を新設し、`setup()`（v1）と併存させ段階移行する。
